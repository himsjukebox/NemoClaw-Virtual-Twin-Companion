NemoClaw turns plain-English design goals into real, downloadable 3D drone parts. Describe what you want ("a 3-blade propeller with 100 mm diameter", "a hexacopter frame with long arms"), and a multi-agent pipeline generates the parametric geometry, validates it against engineering rules grounded in your own reference PDFs, and renders an interactive 3D model you can download as STEP or STL.

## вңЁ Key Features

- **Conversational CAD** вҖ” natural-language prompts become parametric 3D geometry.
- **Multi-part / polymorphic** вҖ” generates three component families: **chassis**, **propeller**, and **motor mount**. The Design Agent automatically picks the right one from your prompt.
- **Configurable rotor count** вҖ” chassis supports `arm_count` 3вҖ“8 (tricopter, quadcopter, pentacopter, hexacopter, octocopter).
- **Real geometry, not a stub** вҖ” the CAD Tool executes [CadQuery](https://cadquery.readthedocs.io/) to produce actual `.STEP` (BREP solid) and `.STL` (mesh) files.
- **Deterministic physics/engineering layer** вҖ” computes All-Up Weight, Thrust-to-Weight Ratio, payload margin, flight time estimate, structural load check, and disk loading from geometry + material + mission inputs. Zero LLM dependency вҖ” all math is pure Python.
- **Mission-aware validation** вҖ” the Validator issues PASS/FAIL based on physics results against use-case targets (racing, cinematography, delivery, mapping), not just parameter ranges.
- **Material selection** вҖ” PLA, ABS, carbon fiber, aluminum вҖ” each with real density and yield strength driving mass and structural calculations.
- **Multimodal RAG grounding** вҖ” the Validator Agent consults your engineering PDFs, including **text and image/diagram captions** generated by an NVIDIA vision model.
- **Iterative self-correction** вҖ” if validation fails (including physics gates), feedback loops back to the Design Agent (up to 3 iterations) with specific issues and actionable suggestions.
- **Engineering metrics UI** вҖ” AUW, TWR, Payload Margin, Flight Time, and Disk Loading displayed with targets and pass/fail indicators.
- **Interactive 3D viewer** вҖ” rotate/zoom the result in-browser, with **STEP and STL download buttons**.
- **100% NVIDIA inference** вҖ” all LLM and embedding calls go through NVIDIA NIM via `langchain-nvidia-ai-endpoints`. No OpenAI/Anthropic dependencies. Physics math stays local.

A LangGraph `StateGraph` orchestrates four nodes. They communicate through a shared **whiteboard state** dictionary rather than calling each other directly вҖ” this keeps routing deterministic and every step observable.

### The four agents/tools

| Node | Responsibility | NVIDIA service |
|------|----------------|----------------|
| **Design Agent** (`main.py`) | Chooses the component type, emits parametric values, material selection, and mission profile as JSON | ChatNVIDIA (Llama 3.1 70B) |
| **CAD Tool** (`tools/cad_tool.py`) | Injects parameters into the matching CadQuery template and runs it to produce STEP + STL | вҖ” (local CadQuery execution) |
| **Physics Analysis** (`main.py` + `tools/physics_engine.py`) | Computes engineering metrics (AUW, TWR, payload margin, flight time, structural load, disk loading) deterministically | вҖ” (pure Python, no inference) |
| **Validator Agent** (`main.py`) | Range-checks parameters, applies the deterministic physics gate, and reasons over RAG context, returning PASS/FAIL + score | ChatNVIDIA + RAG Engine |

### Whiteboard state

| Key | Type | Description |
|-----|------|-------------|
| `user_prompt` | `str` | The user's natural-language design goal |
| `component_type` | `str` | `chassis`, `propeller`, or `motor_mount` |
| `design_parameters` | `dict` | Component-specific numeric parameters |
| `material` | `str \| None` | Selected frame material (PLA, ABS, carbon_fiber, aluminum) |
| `mission_profile` | `dict \| None` | `{payload_mass_kg, use_case, target_flight_time_min}` |
| `frame_volume_m3` | `float \| None` | Chassis solid volume from CAD (or estimated) |
| `engineering_metrics` | `dict \| None` | Full physics analysis results (AUW, TWR, margins, etc.) |
| `cad_output_paths` | `list[str]` | Absolute paths to the generated STL + STEP files |
| `validator_verdict` | `str` | `PASS` or `FAIL` |
| `validator_score` | `float` | Confidence score 0.0вҖ“1.0 |
| `validator_feedback` | `str` | JSON with issues + reasoning (fed back on retry) |
| `iteration_count` | `int` | DesignвҶ’validate cycles completed (max 3) |
| `agent_trace` | `list[dict]` | Chronological log of every node action (observability) |
| `error` | `str \| None` | Error message if any step fails |

## рҹ“ҡ Multimodal RAG

The Validator Agent grounds its verdicts in **your own reference documents**:

1. Drop PDFs into `data/` (drone design guides, material datasheets, standards).
2. On first run the RAG Engine builds a FAISS vector store from:
   - **Text chunks** (500 chars, 50 overlap) extracted with PyPDF.
   - **Image captions** вҖ” PyMuPDF extracts diagrams/figures, and `meta/llama-3.2-90b-vision-instruct` describes each one. These captions become searchable chunks too.
3. The store is persisted to `data/vectorstore/` and reused on subsequent runs.
4. During validation, the agent retrieves the most relevant chunks and factors them into its reasoning.

To re-ingest after adding documents, delete the store:

```bash
rmdir /s /q data\vectorstore     # Windows
# rm -rf data/vectorstore        # Linux/macOS
```

The pipeline is **degraded-mode tolerant**: if there are no PDFs, the vision model is unavailable, or the API is unreachable, validation still runs using built-in range/structural rules.

## рҹҡҖ Getting Started

### Prerequisites

- Python 3.10+
- A free NVIDIA API key from [build.nvidia.com](https://build.nvidia.com/)

### Install

```bash
git clone https://github.com/himsjukebox/NemoClaw-Virtual-Twin-Companion.git
cd NemoClaw-Virtual-Twin-Companion

python -m venv venv
venv\Scripts\activate            # Windows
# source venv/bin/activate       # Linux/macOS

pip install -r requirements.txt
```

> Note: `cadquery` (with the OpenCASCADE kernel) and `pyvista[jupyter]` (with the trame stack for the interactive viewer) are large downloads вҖ” first install may take a few minutes.

### Configure your API key

Copy the example env file and add your key:

```bash
copy .env.example .env           # Windows
# cp .env.example .env           # Linux/macOS
```

Then edit `.env`:

```env
NVIDIA_API_KEY=nvapi-your-key-here
```

> The `.env` file is gitignored and must never be committed.

### Run

```bash
# Web UI (recommended)
streamlit run app.py

# Or run the pipeline directly from the CLI
python main.py
```

Open the Streamlit URL, type a goal like *"Design a tricopter frame with long arms"* or *"Make a 3-blade 5-inch propeller"*, and watch the pipeline generate, validate, and render the part.

## вҡҷпёҸ Physics & Engineering Layer

The physics layer computes real engineering metrics from geometry, material, and mission inputs вҖ” entirely in deterministic Python with no LLM calls:

| Metric | What it tells you |
|--------|-------------------|
| **All-Up Weight (AUW)** | Total assembled drone mass (frame + motors + battery + payload) |
| **Thrust-to-Weight Ratio (TWR)** | Can it fly? Racing needs вүҘ4.0, cinematography вүҘ2.0 |
| **Payload Margin** | How much extra weight it can carry beyond the target payload |
| **Flight Time** | Estimated hover endurance from battery capacity and power draw |
| **Structural Load Check** | Will the arms break under motor thrust? (bending stress vs. yield) |
| **Disk Loading** | Thrust per rotor area вҖ” efficiency and stability indicator |

### Configurable via `config/physics.yaml`

All constants, material tables, motor specs, battery options, and use-case targets are externalized to YAML вҖ” tune engineering assumptions without touching code:

```yaml
materials:
  PLA:           { density: 1240.0, yield_strength: 50.0e6 }
  carbon_fiber:  { density: 1600.0, yield_strength: 600.0e6 }
use_cases:
  racing:         { target_twr: 4.0, default_flight_time_min: 5.0 }
  delivery:       { target_twr: 2.0, default_flight_time_min: 15.0 }
```

## вҡ пёҸ Known Limitations / Roadmap

- **PropellerвҖ“arm clearance** and cross-component compatibility not yet modeled
- **Center of gravity, vibration/resonance** analysis not included
- **Advanced aerodynamics** (blade element theory, drag coefficients) вҖ” current flight time is a hover-only estimate
- **Multi-battery / hybrid power** configurations not supported

**Planned next:**
- NVIDIA Riva integration for a hands-free **Speech-to-CAD** workflow (voice input) and spoken result feedback (TTS)
